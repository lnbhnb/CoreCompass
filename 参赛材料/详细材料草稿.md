## 1. 目标用户与应用场景

**目标用户：** 高校学生团队（3-5 人），尤其是参加课程设计、毕业设计、创新竞赛的本科生。

**应用场景：**
- 拿到一个月以上的大课题，不知道如何拆解
- 团队成员对"做到什么程度算完成"理解不一致
- 进度延期时只能靠口头追问，没有客观数据支撑砍需求决策
- 里程碑验收流于形式，没有客观产物校验

## 2. 真实痛点与解决方案

### 痛点
1. **AI 聊天工具只附和不把关**：ChatGPT/豆包等通用 LLM 顺着用户说话，不验证用户是否真的产出
2. **学生不会主动重规划**：延期后习惯性"加夜班"，而不是冷静砍需求
3. **被动响应式协作**：所有协作工具都靠用户主动查询，没人主动催

### 解决方案：CoreCompass 校园项目"伪需求"粉碎机
基于"真实状态机 + LLM"的混合架构，三大突出点对应三大痛点：

| 痛点 | 突出点 | 核心机制 |
|---|---|---|
| AI 盲信 | ① 硬验收 | 多类型文件结构化校验 + locked 机制 |
| 不会重规划 | ② 动态重算 | 产能缺口公式 + LLM 砍需求提案 + 铁律校验 |
| 被动协作 | ③ 主动打扰 | APScheduler 定时扫描 + 飞书 webhook |

## 3. 核心功能（三突出点详述）

### 突出点① 硬验收（反"AI 盲信"）
**机制：** 每个里程碑声明 `expected_artifact_type`（sql / code / md / json / yaml），上传产物后由对应校验器做结构化校验，不通过则里程碑 `locked`，无法进入下一阶段。

**校验器：**
- SQL：≥2 表、含 PRIMARY KEY、含 FOREIGN KEY
- Markdown：≥500 字、含 H2 标题、无 TODO 标记
- Code：AST 解析、含函数/类定义、非空
- JSON/YAML：语法可解析、schema 结构合理

**铁律：** locked 里程碑的前序依赖必须通过才能解锁。即使 LLM 也不能跳过。

### 突出点② 动态重算（GPS 式重规划）
**机制：** 当检测到任务逾期或 deadline 临近，触发重规划：
1. **规则层**：计算产能缺口 `gap = 剩余工作量 - 剩余天数 × 团队人数 × 0.6`
2. **LLM 层**：把缺口和任务列表喂给豆包，生成砍/降级提案
3. **铁律层**：拒绝砍 `core` 任务，强制砍最高工时的 `optional`

**类比：** 就像 GPS 检测到堵车会重新规划路线，CoreCompass 检测到进度偏航会重新规划任务。

### 突出点③ 主动打扰（反"被动响应"）
**机制：** APScheduler 定时扫描逾期任务，按项目聚合后通过飞书 webhook 主动推送。

**触发时机：**
- 定时：默认每 60 分钟扫描一次
- 手动：Demo 主路径 `/api/notify/test`
- 即时：重规划应用后立即推送

## 4. 技术路线

### 架构图（文字描述）
```
浏览器（Alpine.js）
    ↓ HTTP
FastAPI（路由 + 中间件）
    ├── 项目服务（创建 + LLM 初始拆解）
    ├── 校验服务（5 类文件结构化校验）
    ├── 重规划服务（规则 + LLM + 铁律）
    ├── 通知服务（飞书 webhook + APScheduler）
    └── 状态机（任务/里程碑/项目确定性转移）
    ↓
SQLite（项目 / 任务 / 里程碑 / 校验记录 / 通知日志）
    ↓
火山引擎豆包 LLM（带重试 + JSON 修复 + fallback）
```

### 关键算法
1. **状态机**：`TaskStatus` 五态（planned/doing/done/overdue/cut），转换由 `transition_task()` 强制约束，非法转换抛 `InvalidTransition`
2. **产能公式**：`gap = Σ(est_effort_days × 未完成率) - remaining_days × team_size × 0.6`，效率系数 0.6 反映校园团队实际产出
3. **LLM JSON 修复**：`json_repair` 库兜底 + 正则提取代码块 + fallback 模板
4. **飞书签名**：HMAC-SHA256，timestamp + secret 拼接

## 5. 创新点

| 创新 | 与现有工具的差异 |
|---|---|
| 反 AI 盲信 | Notion/语雀只存文档不校验；通用 LLM 只对话不把关。CoreCompass 强制校验真实产物 |
| GPS 式重算 | Jira/Teambition 只显示逾期不主动决策；CoreCompass 自动算缺口 + LLM 提案 + 铁律保底 |
| 主动打扰 | 飞书/钉钉的提醒依赖人触发；CoreCompass 的 Agent 主动巡检并推送 |

## 6. 实际效果

**Demo 三段式（详见录屏）：**
1. 输入"校园二手交易平台 / 30 天 / 3 人" → 4 里程碑 + 6 周任务计划
2. 上传空 .sql → "缺少外键，进度锁定"；上传合规 .sql → 通过解锁
3. 模拟偏航 → 重规划缺口 +12 人天 → LLM 提案砍 optional → 看板刷新
4. 飞书群几秒内收到 Agent 催办消息

**测试覆盖：** 29 个单元/集成测试全绿，覆盖状态机、校验、重规划、通知、E2E 闭环。

## 7. 推广落地可行性

- **学生侧**：零安装（浏览器访问），5 分钟上手
- **教师侧**：可作为课程设计过程管理工具，查看团队真实进度
- **扩展性**：校验器可插拔（新增类型只需实现 `validate_xxx`）；LLM 可替换（任何兼容 OpenAI 协议的模型）
- **成本**：SQLite 单文件、豆包 LLM 按调用计费、飞书 webhook 免费
